```markdown
一．基本规范
 1.对于不同系统/模块之间的交互，首先要确定彼此之间的API通讯协议：出于安全考虑使用外网之间接口通信使用https，内部之间接口通信使用http；
 2.http请求需要明确请求方式：使用GET、POST，避免造成歧义，当请求参数大于三个的时候，需要使用post请求方式，不建议使用form方式；
 3.调用方需要给出请求量、高峰时段、QPS 等相关指标；
 4.接口的路径需要体现出来自于哪个service，例如：order 代表订单service ；
 5.接口的路径中需要体现出版本号，有利于多版本管理、维护和灰度发布，例如：v1、v2、v3；
 6.接口命名建议采用系统/模块名称+版本号+具体接口名称方式，例如：/order/v1/orderInfo/id/100
 7.对于数据量大、有特殊要求的接口要定义好条件限制规则，例如：分页、排序、筛选等，/order/v1/orderInfo?page=0&pageSize=50；
 8.请求参数：双方明确公共参数、必选参数、可选参数，一定做好参数校验，当必选参数为空时，请求拒绝；可选参数为空，双方要约定好处理方式；
 9.接口返回值：调用方需要哪些字段，接口就返回哪些字段，不要提供太多用不到的字段，影响接口网络传输速率
 10.返回值数据结构一定基于以下格式（可以根据实际情况基于此结构进行扩展）：
{
    sucess：布尔值（表示是否成功）
    code：响应值(http的响应值和自定义响应值)
    message：响应信息(http响应值对应的响应信息和自定义的响应信息)
    data：{
        数据
    }
}
 http的响应值和响应信息一定遵循官方标准，不可私自乱定篡改原响应值和响应信息的意义；
 11.保证接口健壮性：保证不因为传入可选参数加减、返回值加减导致接口异常；
 12.接口数量建议越少越好，能合并的接口尽量合并到一起；
 13.接口调用调用方可以增加 traceId,放在http header中，有利于问题追踪，生成traceId要避免并发条件下冲突，如果在服务端添加traceId，推荐使用
uid-generator（uid-generator/README.zh_cn.md at master · baidu/uid-generator · GitHub）；
 14.当前版本接口优化，尽量保证请求参数和返回值不变，如果无法保证，评估影响范围并告知相关调用方，此时建议升级接口版本；
 15.接口版本升级，避免零散升级，造成版本混乱。
二．接口文档
 1.接口提供者一定要提供接口文档
 包括（不限于）：接口地址、接口版本、请求方式、请求参数、响应参数、QPS、TPS
 2.请求参数
 请求参数一定要明确哪些是必选，哪些是可选，必选参数一定做好参数校验，每个请求参数都要给出中文含义，枚举参数一定要给出所有枚举值及其含
义。
 3.返回值
 返回值建议返回字段保持一致（比如：list中的元素字段和detail中的返回值属性一致），必须给出每个字段的中文含义、响应值和响应信息
三．安全性
 1.新的项目/模块/service需要使用Jackson，鉴于旧的项目很多使用FastJson，出于安全性考虑需要升级FastJson到1.2.75以上版本；
 2.接口一定要鉴权（gateway层），禁止不经过同意随意调用，很多接口这点做的不好；
 3.对于访问量大的接口，调用方和提供方要根据具体业务内容给出合理的每天/小时/分钟访问次数，不可以每天无限次调用，预防数据库压力过大影响正
常业务；如果接口提供方无法满足调用方，要给出具体理由，并做接口性能升级，满足调用方合理调用需求；
 4.对于高峰期可能影响线上业务的接口请求，要做好限流、熔断、降级，做到可配置化；
 5.敏感数据一定要脱敏，即使内部间传递也不可以明文，比如：密码、手机号、身份证、支付账号、身份证号等；
 6.幂等性保证：对于部分接口存在重试、多次发送、存在网络故障等因素，需要结合具体业务要考虑接口的幂等性，不同业务有不同处理方案，双方要
保证好，例如：可以利用订单号、id、token、状态机等保证；
 7.签名设计：使用非对称加密；
 8.一定要对数据的合法性进行校验，可进行常规性校验和业务校验；
 9.接口跨域问题以及安全性要提前规划：跨域问题要区分是CORS还是CSRF，对于CSRF属于跨域安全攻击。
```
